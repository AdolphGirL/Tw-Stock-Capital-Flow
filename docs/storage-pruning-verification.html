🔧 Tw-Stock-Capital-Flow

Layer 1
原始快照
YYYYMMDD.json
每日 ~500KB–1MB
保留 7 天 ≈ 3.5–7 MB
Layer 2
SQLite 歷史
四張歷史表
每日 ~300–500 rows
保留 365 天 ≈ 40–80 MB
Layer 3
分析快取
bootstrap_cache_*.json
每份 ~50–100 KB

🔁
觸發時機與流程
三層清理皆整合在 SyncManager 與 main.dart 的 initialize() 內
ℹ️
只在新交易日資料成功寫入（syncResult.saved == true）時觸發清理。 週末、假日、重開 App 但當日資料已存在 → 不觸發，不影響效能。
🌐
SyncManager.syncTodayData()
抓取上市 / 上櫃資料，判斷是否為新交易日
🗂️
StorageService.pruneOldSnapshots(keepCount: 7) 新交易日

🗃️
CategoryHistoryRepository.pruneOldHistory(keepDays: 365) 新交易日
四張 SQLite 表單一事務刪除 365 天前舊紀錄（Layer 2）
💾
BootstrapAnalyzer.analyzeAsync() → saveBootstrapCache()
無快取時執行計算並儲存今日分析結果
🧹
StorageService.pruneOldBootstrapCaches(keepCount: 3) 每次計算後
刪除超出 3 份的舊分析快取檔（Layer 3）

Layer 1 驗證：原始快照清理
確認 daily_data/ 目錄中的 YYYYMMDD.json 不超過 7 個
1
連接裝置並取得應用目錄路徑
Android 使用 adb，iOS 使用 Xcode Device Explorer
bash — Android 
# 替換 <package_name> 為你的 App bundle id
adb shell run-as <package_name> ls files/daily_data/
2
模擬多日快照（不需等待真實交易日）
手動複製假日期檔案到裝置，製造 8+ 筆快照情境
bash 
# 在裝置上複製現有快照，偽造多個舊日期
adb shell run-as <package_name> \
  cp files/daily_data/20260607.json files/daily_data/20260501.json
# 重複多次製造 8 個以上的 YYYYMMDD.json
3
重新啟動 App，觀察 Debug Console
新交易日 sync 成功後，應看到以下 log 輸出
[SyncManager] 資料同步成功: 20260607
[StorageService] 🗑️ 已清理舊快照: 20260501.json
[StorageService] 🗑️ 已清理舊快照: 20260502.json
[StorageService] 快照清理完成，保留最近 7 天，刪除 N 筆
4
確認目錄最終狀態
目錄中只應剩下最新 7 個 YYYYMMDD.json
bash 
adb shell run-as <package_name> ls -la files/daily_data/ \
  | grep -E '^[0-9]{8}\.json'
# 預期輸出：恰好 7 行（最新 7 個交易日）
✅
通過條件：目錄中 YYYYMMDD.json 檔案數量 ≤ 7， 且最新 5 個日期的快照完整存在（確保計算引擎不受影響）。

Layer 2 驗證：SQLite 歷史清理
確認四張歷史表中不存在 365 天前的紀錄
1
觀察 Debug Console log 輸出
新交易日成功 sync 後，應看到以下訊息
[CategoryHistoryRepository] SQLite 歷史清理完成，cutoff=20250607，共刪除 N 筆舊紀錄
⚠️
若 App 運行未滿 365 天，N = 0 為正常結果，表示無舊紀錄需刪除。
2
取出 SQLite 檔案進行 SQL 驗證
使用 DB Browser for SQLite 開啟資料庫檔案
bash — 取出 db 檔 
adb pull \
  /data/data/<package_name>/databases/app_database.db \
  ./app_database.db
3
執行 SQL 驗證查詢
在 DB Browser 的「Execute SQL」分頁中逐一執行
sql 
-- 四張表的最舊紀錄日期（應皆 >= 365 天前）
SELECT 'category_history'  AS tbl, MIN(trade_date) AS oldest FROM category_history
UNION ALL
SELECT 'mainstream_history', MIN(trade_date) FROM mainstream_history
UNION ALL
SELECT 'lifecycle_history',  MIN(trade_date) FROM lifecycle_history
UNION ALL
SELECT 'rotation_history',  MIN(trade_date) FROM rotation_history;

-- 確認無超期紀錄（應回傳 0）
SELECT COUNT(*) AS expired_rows
FROM category_history
WHERE trade_date < date('now', '-365 days');
✅
通過條件： expired_rows = 0，且四張表的 oldest 日期均不早於 365 天前。
3
Layer 3 驗證：Bootstrap 分析快取清理
確認 bootstrap_cache_*.json 檔案不超過 3 個
1
觀察 Debug Console log 輸出
[StorageService] 快取清理完成，保留最近 3 份，刪除 N 筆
2
確認目錄中快取檔案數量
bash 
adb shell run-as <package_name> \
  ls files/daily_data/ | grep bootstrap_cache_
# 預期輸出：最多 3 行
✅
通過條件：bootstrap_cache_*.json 檔案數量 ≤ 3。
🛡️
核心邏輯安全驗證
確認清理後所有核心功能正常運作
功能 需求資料量 保留量 安全緩衝 狀態
BootstrapAnalyzer.analyzeAsync() 最近 5 天快照 7 天 +2 天 ✅ 安全
MainstreamEngine / LifecycleEngine 最近 3 天快照 7 天 +4 天 ✅ 安全
SubCategoryPage 歷史趨勢圖 15 天 SQLite 365 天 +350 天 ✅ 安全
離線防禦模式 tryGetAnyLatestCache() 最少 1 份快取 3 份 +2 份 ✅ 安全
📋
功能迴歸測試步驟：
1. 清理後重新啟動 App → 確認主畫面正常顯示所有板塊數據
2. 進入任意板塊 → 點擊細分板塊 → 確認歷史趨勢折線圖正常渲染
3. 斷開網路後重啟 → 確認離線模式橫幅出現且顯示歷史數據
